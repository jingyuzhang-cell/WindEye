# 在线协同治理社区报告实施设计

## 1. 目标

在不改变外部接口协议的前提下，重构 `POST /api/v1/governance/compliance-report` 的内部生成逻辑，使其从当前的轻量拼接式报告，升级为“在线版 `new_report` 社区报告引擎”。

外部保持不变：

- 请求入口不变：`/api/v1/governance/compliance-report`
- 请求字段不变：`query`、`seedNames`、`seedIds`、`maxHop`、`maxPathLength`、`exportFormats`、`reportOptions`
- 响应字段不变：`exportFiles.docx`、`report.downloadUrl`、`pipelineTrace`、`complianceIndicators`、`governance`

内部改造目标：

- 复用现有 `community-discovery` 和 `risk-paths`
- 用 `backend/new_report` 的“分视角报告生成逻辑”替换 `_build_compliance_report()`
- 在没有离线预计算社区资产的情况下，基于当前主体的局部子图临时构建在线社区报告


## 2. 改造边界

本次改造只替换第 5 个 API 的报告生成核心，不改动以下接口协议：

- `POST /api/v1/graph/search-all`
- `POST /api/v1/graph/expand/{node_id}`
- `POST /api/v1/governance/community-discovery`
- `POST /api/v1/governance/risk-paths`
- `POST /api/v1/governance/compliance-report`

本次改造也不引入以下能力：

- 不在线运行全量 GNN / Leiden
- 不在线构建全库三视角社区
- 不要求离线社区报告或离线层级映射文件存在


## 3. 总体方案

### 3.1 现有在线链路

当前 `compliance-report` 内部链路已经是：

1. `community-discovery`
2. `risk-paths`
3. `_build_compliance_report()`
4. `docx` 导出

对应入口位于 [governance_routes.py](/D:/Code/WindEye/backend/api/governance_routes.py:1045)。

### 3.2 目标链路

改造后内部链路调整为：

1. 输入归一化
2. 调用 `community-discovery`
3. 调用 `risk-paths`
4. 构建 `OnlineReportContext`
5. 构建三视角局部画像
6. 生成 `new_report` 风格报告
7. 组装旧响应格式
8. 导出 `docx`

### 3.3 关键原则

- 外部协议稳定，内部实现重构
- 在线只做局部子图版社区报告
- 报告生成逻辑从 `route` 中剥离，沉到独立模块
- 支持后续平滑接入离线社区资产，但当前不依赖


## 4. 新增模块设计

建议新增 3 个核心模块，全部放在 `backend/new_report/code/` 下。

### 4.1 `online_report_context_builder.py`

职责：

- 接收 `community-discovery`、`risk-paths`、`subgraph` 原始结果
- 归一化成统一的上下文对象
- 清洗前端/接口返回中的冗余字段
- 提取报告生成最需要的节点、关系、路径、社区摘要

建议核心类：

```python
class OnlineReportContextBuilder:
    def build(
        self,
        seed_nodes: list[dict],
        community_payload: dict,
        risk_payload: dict,
        report_req: ComplianceReportRequest,
    ) -> OnlineReportContext:
        ...
```

建议输出数据结构：

```python
@dataclass
class OnlineReportContext:
    seed_nodes: list[dict]
    subject_name: str
    subgraph_nodes: list[dict]
    subgraph_edges: list[dict]
    communities: list[dict]
    seed_community_id: int | None
    entity_community_map: list[dict]
    community_graph: dict
    risk_paths: list[dict]
    community_risk_paths: list[dict]
    summary: dict
    warnings: list[str]
```

### 4.2 `online_perspective_builder.py`

职责：

- 把局部上下文拆成三种在线视角
- 不跑全局社区算法，只做“当前主体相关”的视角聚合

建议核心类：

```python
class OnlinePerspectiveBuilder:
    def build(self, context: OnlineReportContext) -> PerspectiveBundle:
        ...
```

建议输出：

```python
@dataclass
class PerspectiveBundle:
    responsibility: dict
    violation: dict
    regulatory: dict
    cross_links: list[dict]
```

### 4.3 `online_community_report_engine.py`

职责：

- 复用 `new_report` Step2 的写法生成在线报告
- 产出 markdown、recommendations、governance actions、指标评分
- 最终向旧 response 结构映射

建议核心类：

```python
class OnlineCommunityReportEngine:
    def generate(
        self,
        context: OnlineReportContext,
        bundle: PerspectiveBundle,
        report_req: ComplianceReportRequest,
    ) -> dict[str, Any]:
        ...
```


## 5. 三视角在线构建逻辑

### 5.1 责任方视角

从以下信息中抽取：

- `seedNodes`
- `subgraph.nodes` 中的 `COMPANY / PERSON / PFUND / SECURITY / PFCOMPANY`
- `entityCommunityMap`
- 与种子主体直接相关的控制、任职、投资、关联关系

建议输出字段：

- `coreSubjects`
- `keyPersons`
- `relatedEntities`
- `responsibilityRelations`
- `communitySummary`

### 5.2 违规行为视角

从以下信息中抽取：

- `riskPaths`
- `communityRiskPaths`
- `subgraph.nodes` 中的 `EVENT / Action / RiskFeature / RiskFactor`
- `riskPaths[].pathDescription`

建议输出字段：

- `keyEvents`
- `keyActions`
- `riskFeatures`
- `topRiskPaths`
- `propagationSummary`

### 5.3 监管 / 法规视角

从以下信息中抽取：

- `subgraph.nodes` 中的 `Law / Section / Chapter / REGULATOR`
- `riskPaths` 中涉及法规或监管机构的路径
- `communityRiskPaths` 中涉及法规支撑的社区链

建议输出字段：

- `regulators`
- `laws`
- `sections`
- `regulatoryTriggers`
- `complianceBasis`

### 5.4 跨视角连接

不做全量层级映射，只在当前主体局部图上表达：

- 主体 -> 事件
- 事件 -> 法规
- 主体 -> 法规
- 主体 -> 人员 -> 事件
- 主体 -> 风险特征 -> 法规

建议统一表示为：

```python
{
  "sourcePerspective": "responsibility",
  "targetPerspective": "violation",
  "relationType": "INVOLVED_IN",
  "evidenceNodeIds": [...],
  "evidenceEdgeIds": [...],
  "score": 0.82
}
```


## 6. 报告章节设计

在线报告建议固定输出以下章节：

1. 执行摘要
2. 主体所在社区概览
3. 责任方视角分析
4. 违规行为与风险传导分析
5. 监管机构与法规依据分析
6. 跨视角治理链条
7. 协同治理建议
8. 证据附录

每一章建议分别输出：

- `summary`
- `key_points`
- `evidence`
- `risk_level`

## 8. 改造 `governance_routes.py` 的方式

### 8.1 保留不动

以下逻辑保持：

- `community_discovery()`
- `risk_paths()`
- `docx` 导出逻辑
- 下载文件接口

### 8.2 替换点

当前 `_build_compliance_report()` 位于 [governance_routes.py](/D:/Code/WindEye/backend/api/governance_routes.py:531)。

建议改成：

```python
def _build_compliance_report(req: ComplianceReportRequest) -> dict[str, Any]:
    context = OnlineReportContextBuilder().build(...)
    bundle = OnlinePerspectiveBuilder().build(context)
    result = OnlineCommunityReportEngine().generate(context, bundle, req)
    return result
```

`route` 本身只负责：

- 编排上游接口
- 收集 request
- 触发报告引擎
- 导出 docx
- 回填 response

## 10. 风险与控制

### 风险 1：在线报告不等于离线全局社区报告

说明：

- 在线版只基于当前主体局部子图
- 不能声称其等价于全库三视角聚类结果

控制：

- 在 `pipelineTrace` 标注 `engineMode=online_new_report`
- 在报告中注明“基于当前主体局部社区结果生成”

### 风险 2：上下游字段不稳定

说明：

- `community-discovery` 和 `risk-paths` 返回字段若变动，会影响新引擎

控制：

- 在 `OnlineReportContextBuilder` 中做字段归一化
- 不让 `route` 直接依赖下游字段细节

### 风险 3：报告质量不稳定

说明：

- 如果直接用原始 JSON 拼接，报告会再次退化成“摘要壳”

控制：

- 强制采用章节模板
- 先结构化，再渲染文本


## 11. 验收标准

满足以下条件视为第一阶段完成：

1. `/api/v1/governance/compliance-report` 的 request/response 不变
2. 内部不再直接使用旧版 `_build_compliance_report()` 拼接逻辑
3. 新引擎能基于单一公司主体生成 8 章节 markdown 报告
4. `exportFiles.docx`、`report.downloadUrl` 仍然可用
5. `pipelineTrace` 能显示 `engineMode=online_new_report`
6. 在没有离线社区资产的情况下可独立运行
