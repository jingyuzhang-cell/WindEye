## 1. 引言

### 1.1 目的

通过本次系统测试，验证 WindEye v1.0 各功能模块是否满足《v1.0 需求分析文档》和《v1.0 概要设计文档》《v1.0 详细设计文档》中定义的功能和性能要求，尽可能发现程序设计和实现中的错误与缺陷，确保系统达到可交付的质量标准。

具体目标包括：
- 验证四层知识图谱（Subject → Event → Feature → Regulation）的构建、查询和可视化功能是否完整正确
- 验证 DRA-MA（Dynamic Reasoning and Multi-Agent）多智能体协同治理引擎的风险分析能力
- 验证系统在并发、大数据量场景下的性能表现满足非功能性需求指标
- 验证认证、授权、审计等安全机制的完整性和可靠性

### 1.2 范围

本次测试覆盖 WindEye 系统的以下模块和功能：

| 模块 | 覆盖内容 | 前端路由 |

## 2. 测试对象

### 2.1 产品架构

WindEye 采用四层架构：

```
数据采集层 → 知识图谱层（Neo4j 四层KG）→ 多智能体推理层（DRA-MA）→ 应用层（React 前端）
                  ↑                                                              |
                  +——————— 人机协同反馈闭环 —————————————————————————————————————+
```

**四层知识图谱模型**：

| 层级 | 名称 | 实体类型 |
|------|------|---------|
| Layer 1 | Subject 主体层 | Company, Person, PFCompany, PFund, Security |
| Layer 2 | Event 事件层 | Event, SubEvent, Time, Regulator |
| Layer 3 | Feature 特征层 | RiskFeature, RiskFactor |
| Layer 4 | Regulation 法规层 | Regulation, Law, Action, PartyWithResponsibility |

**三大模块**：

1. **模块A（知识图谱构建）**：数据采集（CrawlOrchestrator 5 阶段）→ 实体抽取（NER 三引擎 + Dify 四阶段 Workflow）→ ETL 管道（PipelineRunner 7 阶段）→ Neo4j
2. **模块B（知识图谱展示）**：Neo4j → 16 个图谱 API → GraphAnalytics 分析引擎（7 种社区发现算法、中心性、环路检测）→ G6 四层可视化
3. **模块C（协同治理）**：用户输入 → 意图路由 → Graph Tab（DRAEngine 4 层 × 5 阶段）/ Risk Tab（RiskAnalysisEngine 7 阶段）→ SSE 流式输出 → G6 图谱 + 风险报告面板

## 4. 测试方法与策略

### 4.1 测试策略

总体策略：**黑盒测试为主，白盒测试为辅**。

- **黑盒测试**：基于需求文档和 API 文档，验证各功能模块的输入输出是否符合预期。使用等价类划分、边界值分析、错误推测等方法设计测试用例。
- **白盒测试**：对关键算法（Cypher 生成、5 级实体对齐、图分析算法、Cypher 注入防护）进行代码级逻辑覆盖验证。

### 4.2 测试数据

**基础测试数据**：使用 `sample_data.cypher`（88 个节点、113 条关系，覆盖 3 个企业集团），覆盖四层图谱的全部实体类型和关系类型。

**补充测试数据**：
- 正常业务数据：模拟真实资本市场风险事件的合规数据
- 边界数据：空数据、单节点、超长名称（> 255 字符）、特殊字符（Unicode/CJK）、极大/极小数值
- 异常数据：格式错误 JSON、残缺 Cypher 语句、超大文件（> 100MB）
- 权限测试数据：为 4 种角色（admin/analyst/auditor/readonly）各创建测试用户

### 4.3 测试阶段

#### 4.3.1 单元测试

单元测试在开发实现阶段由开发人员完成，以系统模块为单位进行测试。测试重点在于验证各模块的功能和性能指标，以及对接要求。

本次单元测试主要覆盖以下模块：
- **后端**：CrawlOrchestrator 5 阶段、PipelineRunner 7 阶段、NER 三引擎、EntityResolver 5 级对齐、CypherGenerator、DRAEngine 4 层 × 5 阶段、RiskAnalysisEngine 7 阶段、Auth 认证中间件、RBAC 权限检查
- **前端**：核心组件渲染、Zustand Store 状态管理、SSE 事件处理、API 调用封装

**已有测试基础**：`backend/tests/` 目录下已有 16 个 pytest 文件，覆盖：
- 爬虫配置和工具函数（`test_scrapers.py`、`test_szse_download.py`、`test_download.py`）
- 图 API 查询（`test_graph_search_api.py`、`test_graph_Nhop_api.py`、`test_search_all_api.py`）
- 图分析（`test_graph_analytics_format.py`、`test_graph_statistics.py`）
- 社区发现（`test_community_api.py`）
- 风险路径（`test_risk_path__api.py`）
- 认证（`test_auth_rollout.py`、`test_dev_admin_store.py`）
- SSE 流式（`test_sse_live.py`）
- 意图代理（`test_intent_agent_fallback.py`）
- Neo4j 降级（`test_graph_data_unavailable.py`）

#### 4.3.2 集成测试

由测试人员完成，以系统各模块之间的接口为重点进行测试。重点测试各模块集成后的功能与性能指标。

关键集成测试场景：
- 模块A → 模块B 端到端：数据采集 → ETL → Neo4j 导入 → 图谱查询 → 可视化渲染
- 模块B → 模块C 联动：图谱节点选中 → 传递到 KnowledgeQA → 触发风险分析
- 前端 → 后端 → Neo4j 全链路：页面操作 → API 调用 → Cypher 执行 → 结果返回 → 前端渲染
- 认证 → 权限 → 审计全链路：登录获取 Token → 携带 Token 操作 → 权限检查 → 操作记录审计日志

#### 4.3.3 系统测试

由测试人员完成，在接近真实的运行环境中进行测试。主要验证系统集成后的功能和性能指标，以及稳定性、容错性和可扩展性。

系统测试场景：
- 全业务流程串联：从数据采集到知识图谱构建，再到可视化查询，最后到协同治理分析和报告生成
- 多用户并发：模拟 10 个用户同时登录并进行不同操作
- 长时间运行：连续运行 72 小时，监控内存/CPU/连接池状态
- 异常恢复：模拟 Neo4j/MySQL/Redis 服务宕机后恢复

#### 4.3.4 验收测试

由测试人员完成，对各系统之间的接口为重点进行测试。重点测试各系统集成后的功能和性能指标。

验收测试以用户场景为核心，使用真实的业务问题进行端到端验证：
- 用户场景 1：监管人员查询"鑫达投资有限公司"的风险传导路径
- 用户场景 2：分析师对"新能源行业"进行合规风险分析
- 用户场景 3：审计员查看系统操作日志并导出报告

#### 4.3.5 指标验证

由测试人员完成，主要针对集成平台完成集成测试后是否满足用户提出的指标要求。

### 4.4 测试内容分类

#### （1）用户界面测试（UI 测试）

检查系统用户界面的正确性、容错性和友好性。包括：
- 页面布局与设计稿一致性
- 交互控件功能正确性（按钮、输入框、下拉菜单、弹窗）
- 表单校验与错误提示
- 响应式布局兼容性（≥ 1366×768）
- 加载状态与空状态展示
- G6 图谱交互（缩放、拖拽、点击、双击展开、Tooltip）

#### （2）功能测试

验证系统是否满足需求定义的功能。在本次测试中主要测试各模块的功能是否达到需求文档中的要求。

重点测试项：
- API 端点输入输出正确性（对照 [R4] API 文档）
- SSE 事件流完整性和顺序正确性
- 业务逻辑分支覆盖（正常流程、异常流程、边界条件）
- 数据一致性（Neo4j 图谱数据与入库源数据一致）

#### （3）性能测试

测试系统响应时间、吞吐量、数据处理速率及时性等。主要关注数据采集环部分。

根据 [R5] 文档定义的性能指标执行测试。

#### （4）强度测试

通过增大数据量和并发量，确认系统在极限条件下的表现是否达到要求。

场景：
- 大数据量图谱渲染（1000+ 节点、5000+ 节点）
- 高并发 API 请求（50/100/200 QPS）
- 长时间持续采集任务（连续采集 100+ 页）

#### （5）可靠性测试

通过长时间运行和异常注入，确认系统是否具备容错能力、故障恢复能力。

场景：
- 72 小时连续运行监控
- Neo4j 连接中断后的自动重连
- Redis/MySQL 服务不可用时的降级处理
- 爬虫网络超时重试机制
- 单文件解析失败不阻塞管道其余文件

### 4.5 测试方法

针对各项测试，明确主要指标的定义、相关测试数据，以及组合测试环境下的综合测试方法。

主要测试方法包括：
- **等价类划分**：对输入参数（搜索关键词、日期范围、页码）划分有效和无效等价类
- **边界值分析**：对数值型参数（limit、depth、max_pages）测试边界值（0、1、最大值、最大值+1）
- **错误推测**：基于 .learnings/ERRORS.md 中的已知错误设计回归测试用例
- **场景法**：模拟真实用户业务流程的端到端测试
- **正交实验法**：对多参数组合的 API 端点（如 /search 含 layer、nodeType、relType 等多参数）使用正交表

### 4.6 测试评价

- 从设计的角度评价程序功能的正确性、性能的可靠性。
- 从使用的角度评价人机交互界面的友好性和易用性。

## 7. 测试过程

### 7.1 测试流程

```
准备阶段 → 执行阶段 → 回归阶段 → 报告阶段
```

**准备阶段**：
- 搭建测试环境（硬件、软件、网络）
- 部署测试版本到测试环境
- 导入测试数据（sample_data.cypher）
- 创建测试用户（admin/analyst/auditor/readonly）
- 编写和验证自动化测试脚本

**执行阶段**：
- 按照《测试大纲》执行测试项
- 记录测试结果到测试记录表
- 发现问题记录到问题汇总表
- 每日汇总测试进度

**回归阶段**：
- 开发人员修复缺陷
- 测试人员对修复进行回归验证
- 确认无新增缺陷

**报告阶段**：
- 汇总全部测试结果
- 形成测试报告
- 给出测试结论

### 7.2 测试通过标准

| 标准 | 判定条件 |
|------|---------|
| P0 关键功能 | 全部通过，0 个未解决缺陷 |
| P1 重要功能 | 通过率 ≥ 95%，无阻塞性缺陷 |
| P2 一般功能 | 通过率 ≥ 90% |
| 性能指标 | P95 响应时间达标率 ≥ 95% |
| 安全测试 | 无高危/严重漏洞 |
| 72h 稳定性 | 无致命故障导致服务不可用 |

### 7.3 缺陷严重等级

| 等级 | 定义 | 响应时间 | 修复时间 |
|------|------|---------|---------|
| **致命** | 系统崩溃、数据丢失、安全漏洞可被远程利用 | 即时 | 24h 内 |
| **严重** | 核心功能不可用、数据错误 | 4h 内 | 48h 内 |
| **一般** | 非核心功能异常、可规避的操作问题 | 24h 内 | 下一迭代 |
| **轻微** | UI 显示问题、文案错误、优化建议 | 记录即可 | 后续版本 |
