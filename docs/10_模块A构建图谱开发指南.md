## 3. 模块 A 是什么

模块 A 负责**把原始数据变成结构化知识图谱**。一句话概括：

> 多源数据采集 → NER/Dify 实体关系抽取 → 六阶段 ETL → Neo4j 四层图谱

**三阶段流水线：**

```
Phase 1: 数据采集               Phase 2: 实体关系抽取          Phase 3: 图谱导入
┌─────────────────────┐    ┌──────────────────────┐    ┌──────────────────────┐
│ CrawlOrchestrator    │    │  双路径并行:           │    │  PipelineRunner       │
│ ① RequirementParser  │ →  │  路径A: NER三引擎      │ →  │  parse → extract      │
│ ② SourceMatcher      │    │    rule/spaCy/LLM     │    │  → link → resolve     │
│ ③ 爬虫执行           │    │  路径B: Dify 四阶段    │    │  → import → index     │
│ ④ QualityAssessor    │    │    subject→event→      │    │                       │
│ ⑤ RetryHandler       │    │    feature→regulation  │    │  输出: 四层知识图谱    │
│                       │    │                       │    │  Subject → Event      │
│  输出: PDF/TXT 文件    │    │  输出: JSONL 三元组    │    │  → Feature → Regulation│
└─────────────────────┘    └──────────────────────┘    └──────────────────────┘
```

**模块 A 的代码分布在两个包中：**

| 包 | 目录 | 职责 |
|----|------|------|
| `data_collection` | `backend/data_collection/` | Phase 1 爬虫 + Phase 2 NER/Dify |
| `kg_construction` | `backend/kg_construction/` | Phase 2 实体链接 + Phase 3 ETL |

## 4. 端到端数据流

以下是一个公告文件的完整旅程，理解了它你就理解了整个模块 A：

```
1. 用户操作
   ├─ 上传 PDF 到 KnowledgeBuild 前端页面
   └─ 或使用智能爬虫采集（Quick/Complex/Template 模式）

2. CrawlOrchestrator (orchestrator.py:94)
   ├─ Stage 1: RequirementParser.parse_quick_mode()  → config dict
   ├─ Stage 2: SourceMatcher.match()   → scraper_configs
   ├─ Stage 3: 爬虫执行 → PDF/TXT 落盘 data_collection/scrapers/data/
   ├─ Stage 4: QualityAssessor.assess() → quality_score
   └─ Stage 5: 触发 ETL

3. PipelineRunner (pipeline_runner.py:48)
   ├─ parse:  PDF → PyMuPDF + OCR → 纯文本
   ├─ extract:
   │   ├─ 路径A: NERModel → 正则/spaCy/LLM → [{mention, type}]
   │   └─ 路径B: DifyClient → POST /v1/workflows/run → JSONL 三元组
   ├─ link:    EntityLinker → 已抽取实体 ↔ Neo4j 已有节点
   ├─ resolve: EntityResolver → 去重 + 合并
   ├─ import:  CypherGenerator → JSONL → MERGE 语句 → Neo4j
   └─ index:   IndexManager → 重建全文索引

4. 最终结果
   Neo4j 中新增节点（如 :Company, :Event, :Law）
   和关系（如 :INVOLVED_IN, :BASED_ON, :SUBJECT_TO）
   前端 KnowledgeGraph 页面可即时查询
```

**关键 URL 映射：**

| 前端页面 | 路由 | 对应后端 API |
|----------|------|-------------|
| 知识图谱构建向导 | `/knowledge-build` | `POST /api/v1/pipeline/run` |
| 智能采集入口 | `/data-collection` | `POST /api/v1/pipeline/crawl/run` (SSE) |
| 四层图谱可视化 | `/knowledge-graph` | `POST /api/v1/graph/search-all` |

```bash
cd backend

# 演示模式（无 Chrome，使用模拟数据）
python -c "
from data_collection.orchestrator import CrawlOrchestrator
import asyncio

async def test():
    from api.crawl_schemas import CrawlTaskRequest, CrawlMode, DataType
    orch = CrawlOrchestrator(demo_mode=True)
    req = CrawlTaskRequest(
        mode=CrawlMode.QUICK,
        data_type=DataType.RISK_EVENT,
        sources=['sse'],
        keywords=['诉讼'],
        max_pages=1,
    )
    async for event in orch.execute(req):
        print(event['event'], event.get('data', {}).get('message', ''))

asyncio.run(test())
"
```

### 6.2 单独测试 ETL 流水线

```bash
# 先确保 data_collection/scrapers/data/ 下有测试文件

python -c "
from kg_construction.etl.pipeline_runner import PipelineRunner

config = get_pipeline_config()
runner = PipelineRunner(config)
# 添加 stage handler
```

## 7. 常见开发任务

### 7.1 新增一个数据源

1. 在 `backend/data_collection/scrapers/` 下新建爬虫文件
2. 在 `__init__.py` 的 `SCRAPER_REGISTRY` 中注册

## 附录：模块 A 关键术语表

| 术语 | 英文 | 含义 |
|------|------|------|
| 爬虫编排器 | CrawlOrchestrator | 五阶段多源数据采集编排引擎 |
| 需求解析器 | RequirementParser | 将用户输入转为结构化爬取参数 |
| 信源匹配器 | SourceMatcher | 将需求匹配到具体爬虫和数据源 |
| 质量评估器 | QualityAssessor | 对爬取结果进行质量评分 |
| 重试处理器 | RetryHandler | 指数退避重试机制 |
| ETL 流水线 | PipelineRunner | 六阶段 ETL 引擎 |
| 实体链接 | EntityLinker | 将抽取的实体关联到 Neo4j 已有节点 |
| 实体消歧 | EntityResolver | 多源实体去重合并 |
| Cypher 生成器 | CypherGenerator | 将 JSONL 三元组转为 Neo4j MERGE 语句 |
| NER 三引擎 | NERModel | rule (正则) / spaCy (统计) / LLM (语义) |
| Dify 客户端 | DifyClient | Dify 工作流 HTTP API 封装 |
| 四层图谱 | Four-Layer KG | Subject → Event → Feature → Regulation |
