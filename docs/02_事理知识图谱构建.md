# 02 — 事理知识图谱构建

## 2.1 模块概述

事理知识图谱构建是整个系统的**知识底座**，涵盖从数据采集、清洗、本体建模、知识抽取到图谱存储的完整链路。它将多源异构数据转化为结构化的四层知识网络，连接"数据"与"智能"。

整体管线：**数据采集 → ETL 清洗 → 本体建模 → 知识抽取(NER+RE) → 实体对齐 → Neo4j 入库 → 索引就绪**

---

## 2.2 数据采集

### 2.2.1 结构化数据源

| 数据源 | 数据类型 | 接入方式 | 关键字段 | 更新频率 |
|--------|---------|---------|---------|---------|
| 工商注册信息 | 企业基础信息 | API 接口 / 离线包 | 统一社会信用代码、企业名称、法人、注册资本、经营范围、股东信息 | 日级增量 + 全量月度 |
| 司法裁判文书 | 诉讼/执行信息 | 爬虫 / API | 案号、当事人、案由、判决结果、执行标的 | 日级 |
| 行政处罚记录 | 违规信息 | API / 爬虫 | 处罚对象、处罚原因、处罚结果、决定日期 | 日级 |
| 知识产权数据 | 专利/商标/软著 | API 接口 | 申请人、专利号、类别、有效期 | 周级 |
| 招投标信息 | 商业活动 | 爬虫 | 招标方、中标方、项目金额、时间 | 日级 |
| 内部业务数据 | 交易/合同/客户 | 数据库同步 / API | 交易流水、合同信息、客户画像 | 实时/准实时 |

### 2.2.2 非结构化数据源

| 数据源 | 数据类型 | 接入方式 | 提取目标 |
|--------|---------|---------|---------|
| 新闻舆情 | 媒体报道 | 爬虫 (RSS/网页) | 涉事主体、事件类型、情感倾向 |
| 社交媒体 | 公开帖文 | API / 爬虫 | 关键人物言论、公众情绪 |
| 研报/公告 | PDF/DOC 文件 | 文件上传 / 爬虫 | 财务指标、重大事项、风险提示 |
| 法律文书 | 裁判文书/行政处罚书 | PDF 解析 | 当事人、案由、判决/处罚结果 |

### 2.2.3 采集方式

**定向爬虫：**

```
能力矩阵：
┌──────────────┬────────────────────────────────────┐
│ 采集策略      │ 全量采集 / 增量采集 / 事件触发采集   │
│ 调度方式      │ Cron 定时 + 任务优先级队列          │
│ 反爬应对      │ 代理 IP 池轮换 + User-Agent 轮换   │
│              │ 请求频率控制 + 验证码识别接口         │
│ 渲染引擎      │ 静态 (requests/httpx) + 动态        │
│              │ (Playwright) 双引擎，按需切换        │
│ 状态监控      │ 采集成功率、延迟、数据量 Dashboard   │
└──────────────┴────────────────────────────────────┘
```

**API 接口同步：**
- 支持 RESTful API、GraphQL 等多种协议
- Token/OAuth 认证机制，支持凭证管理和自动续期
- 接口响应 Schema 校验与异常告警
- 断点续传：同步中断后可从中断点恢复

**离线文件导入：**

| 文件类型 | 解析方案 | 备注 |
|---------|---------|------|
| CSV / Excel | pandas / openpyxl | 支持列映射配置 |
| JSON / XML | 标准库解析 | 支持嵌套结构扁平化 |
| PDF (文本型) | pdfplumber / PyMuPDF | 提取文本 + 表格 |
| PDF (扫描件) | OCR (Tesseract / 云端 OCR API) | 需预处理（去噪、倾斜校正） |
| 图片 | OCR + 布局分析 | 证照类文件识别 |

## 2.3 数据清洗与标准化

### 2.3.1 清洗流水线

```
原始数据 → 去重 → 格式标准化 → 缺失值处理 → 异常值检测 → 实体消歧 → 标准化输出
```

### 2.3.2 关键能力

| 能力 | 说明 | 技术方案 |
|------|------|---------|
| **去重** | 多源数据中同一实体的重复记录 | 关键字段哈希 + 相似度匹配 |
| **实体消歧** | 同名企业/人名区分 | 上下文特征（注册地、行业、关联方）+ 图结构特征 |
| **缺失值处理** | 必填字段缺失的补全或标记 | 规则填充 + LLM 推断（带置信度标记） |
| **异常检测** | 数据格式/值域/逻辑异常 | 规则引擎 + 统计模型 |
| **非结构化抽取** | 从文本中提取结构化字段 | spaCy NER + LLM 信息抽取 |
| **日期/金额标准化** | 统一日期格式、币种转换 | 规则转换 + 汇率服务 |

### 2.3.3 数据质量度量

每批数据入库前需输出质量报告：

| 指标 | 计算方式 | 及格线 |
|------|---------|--------|
| 完整性 | 必填字段非空率 | ≥ 95% |
| 唯一性 | 去重后唯一实体占比 | ≥ 99% |
| 时效性 | 数据从产生到入库的延迟 | ≤ 24h |
| 一致性 | 跨源数据关键字段冲突率 | ≤ 5% |

### 2.3.4 分层存储

- **原始层 (Raw)：** 采集到的原始数据，保留完整快照，不做任何修改
- **清洗层 (Cleaned)：** 经过标准化处理的结构化数据，按实体类型分表存储
- **就绪层 (Ready)：** 已消歧、已对齐的数据，可以直接写入知识图谱

存储介质：PostgreSQL (结构化数据) + MinIO/本地文件系统 (原始文件快照)

## 2.4 本体模型设计 (Ontology)

### 2.4.1 四层整体结构

```
┌──────────────────────────────────────────────┐
│  法规/行动层 (Regulation & Action Layer)      │
│  Node: Regulation, Policy, Action             │
│  Rel:  REGULATES, PRESCRIBES, MITIGATES       │
│  例：《公司法》第20条 → 规制 → 抽逃出资行为      │
└──────────────────┬───────────────────────────┘
                   │ REFINES / CONSTRAINS
┌──────────────────┴───────────────────────────┐
│  特征层 (RiskFeature Layer)                   │
│  Node: RiskFeature, RiskPattern               │
│  Rel:  EXHIBITS, TRIGGERS                     │
│  例：公司A → 表现 → "资金链断裂特征"            │
└──────────────────┬───────────────────────────┘
                   │ DETECTED_FROM
┌──────────────────┴───────────────────────────┐
│  事件层 (Event Layer)                         │
│  Node: Investment, Litigation, Penalty, News   │
│  Rel:  INVESTS_IN, SUES, SANCTIONS, REPORTS   │
│  例：公司A → 被列为被执行人 → 执行案件001        │
└──────────────────┬───────────────────────────┘
                   │ INVOLVES
┌──────────────────┴───────────────────────────┐
│  主体层 (Subject Layer)                       │
│  Node: Company, Person, Institution, Product   │
│  Rel:  CONTROLS, EMPLOYS, GUARANTEES, SUPPLIES│
│  例：张三 → 担任 → 法人代表 → 公司A             │
└──────────────────────────────────────────────┘
```

### 2.4.2 主体层 (Subject Layer)

**Company 实体：**

| 属性 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `uscid` | String | Y | 统一社会信用代码 (唯一标识) |
| `name` | String | Y | 企业全称 |
| `short_name` | String | N | 企业简称 |
| `legal_person` | String | N | 法定代表人 |
| `registered_capital` | Float | N | 注册资本 (万元) |
| `established_date` | Date | N | 成立日期 |
| `status` | Enum | Y | 经营状态：存续/注销/吊销/迁出 |
| `industry` | String | N | 所属行业 (国民经济行业分类) |
| `province` / `city` | String | N | 注册地 |

**Person 实体：**

| 属性 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `name` | String | Y | 姓名 |
| `id_type` | Enum | N | 证件类型 |
| `id_number_hash` | String | N | 证件号哈希（脱敏存储） |
| `gender` | Enum | N | 性别 |

**Institution 实体：**

| 属性 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `name` | String | Y | 机构全称 |
| `type` | Enum | Y | 类型：政府/司法/金融机构/交易所 |

### 2.4.3 事件层 (Event Layer)

| 事件类型 | 关键属性 | 关联主体 |
|---------|---------|---------|
| **Investment** (投资) | 投资金额、比例、时间 | (Company)-[:INVESTS_IN]->(Company) |
| **Employment** (任职) | 职位、入职时间、离职时间 | (Person)-[:EMPLOYED_BY]->(Company) |
| **Litigation** (诉讼) | 案由、诉讼地位、案号、日期 | (Company/Person)-[:INVOLVED_IN]->(Litigation) |
| **Penalty** (处罚) | 处罚类型、原因、金额、机关 | (Company/Person)-[:RECEIVES]->(Penalty) |
| **Guarantee** (担保) | 担保金额、担保类型、起始日期 | (Company)-[:GUARANTEES]->(Company) |
| **FundFlow** (资金流动) | 金额、方向、时间 | (Entity)-[:TRANSFERS_TO]->(Entity) |
| **NewsEvent** (新闻) | 标题、摘要、情感、来源 URL | (Entity)-[:MENTIONED_IN]->(NewsEvent) |

### 2.4.4 特征层 (RiskFeature Layer)

抽象出的风险标签，由规则引擎或 Agent 推理自动挂载。

| 特征类型 | 识别方式 | 示例 |
|---------|---------|------|
| **资金链断裂特征** | 规则：被执行金额 > 净资产 * 80% | "高负债风险" |
| **高管频繁变更** | 规则：12 个月内法定代表人变更 ≥ 2 次 | "治理不稳定风险" |
| **关联交易密集** | 算法：关联交易占比 > 营收 50% | "利益输送嫌疑" |
| **隐性关联** | Graph 算法：共同高管 / 共同股东 / 同址 | "隐性一致行动人" |
| **空壳公司** | 规则：无实际经营地 + 无社保记录 + 认缴制未实缴 | "壳公司风险" |
| **异常资金环流** | 算法：资金 A→B→C→A 闭环检测 | "资金循环流转嫌疑" |

### 2.4.5 法规/行动层 (Action & Regulation Layer)

| 节点类型 | 关键属性 | 示例 |
|---------|---------|------|
| **Regulation** | 法规名称、文号、颁布机关、生效日期、条款内容 | 《公司法》第 20 条：禁止滥用公司法人独立地位 |
| **Policy** | 政策名称、发布机构、发布日 | 《关于加强金融风险防控的指导意见》 |
| **Action** | 处置类型、依据法条、执行部门 | 冻结账户、限制高消费、发函问询 |

## 2.5 知识抽取

### 2.5.1 抽取流水线

```
非结构化文本 → 分句/分段 → NER 实体识别 → RE 关系抽取 → 实体链接 → 写入图谱
```

### 2.5.2 双引擎策略

| 引擎 | 适用场景 | 优势 | 劣势 |
|------|---------|------|------|
| **规则引擎** (spaCy + 词表匹配) | 结构化程度高的文本（裁判文书、工商变更记录） | 速度快、可解释、成本低 | 泛化能力弱 |
| **LLM 引擎** (DeepSeek / GPT) | 非结构化文本（新闻、社交媒体、研报） | 语义理解强、泛化好 | 速度慢、成本高、需 Prompt 调优 |

**切换规则：** 默认使用规则引擎，当规则引擎置信度 < 阈值时，fallback 至 LLM 引擎。系统通过 `NER_MODEL` 环境变量控制：`rule` / `llm` / `hybrid`。

## 2.6 知识融合与对齐

### 2.6.1 实体对齐

将多源数据中指向同一现实实体的节点进行合并：

| 对齐策略 | 适用场景 | 方法 |
|---------|---------|------|
| **精确匹配** | 统一社会信用代码、身份证哈希 | 唯一键匹配 |
| **模糊匹配** | 企业名称（含曾用名、简称） | TfIdf + 余弦相似度，阈值 ≥ 0.85 |
| **图结构匹配** | 无名实体或信息残缺 | 邻居节点相似度 + 图嵌入 |

### 2.6.2 冲突解决

多源数据对同一属性有不同值时：
- **时间戳优先：** 取最新数据
- **权威度优先：** 官方数据源 > 第三方 > 网络采集
- **置信度标记：** 冲突无法解决时，保留多值并标记 `confidence: 0.5`

## 2.7 图谱存储 (Neo4j)

### 2.7.1 存储规范

| 配置项 | 建议值 | 说明 |
|--------|--------|------|
| 节点标签 | 按四层分类 (Subject, Event, Feature, Regulation) | 便于按层查询 |
| 关系类型 | 大写 + 下划线，语义化命名 | 如 `CONTROLS`, `INVESTS_IN` |
| 索引 | 唯一标识字段建索引 | `uscid`, `name`, `id_hash` |
| 分区 | 按行业/地域分区 | 大规模部署时 |

### 2.7.2 对外查询 API

| 接口 | 描述 | 响应时间要求 |
|------|------|------------|
| `/kg/entity/{id}` | 单实体详情 (含 1 跳邻居) | < 200ms |
| `/kg/expand` | 从指定节点 N 跳扩展子图 | < 500ms (3 跳) |
| `/kg/path` | 两节点间最短路径 / 所有路径 | < 1s (5 跳) |
| `/kg/community` | 社区发现 (Louvain / Label Propagation) | < 5s |
| `/kg/search` | 全文检索实体 | < 300ms |
| `/kg/cypher` | 受控 Cypher 直查 (只读) | 取决于查询复杂度 |

## 2.8 系统架构定位

```
┌──────────────────────────────────────────────────────────────┐
│                     知识计算层 (Knowledge Graph)              │
│  ┌─────────────────────────────────────────────────────┐     │
│  │  法规/行动层  │ 法律法规、监管要求、阻断/处置动作       │     │
│  │  特征层      │ 风险标签：资金链断裂、高管频繁变更...    │     │
│  │  事件层      │ 投资、诉讼、处罚、资金流动、新闻舆情     │     │
│  │  主体层      │ 企业、人员、机构、产品、账户             │     │
│  └─────────────────────────────────────────────────────┘     │
│  ┌───────────┐ ┌───────────┐ ┌───────────────────────┐      │
│  │知识抽取    │ │实体对齐/融合│ │ 图存储/查询 (Neo4j)   │      │
│  │(NER/RE)   │ │           │ │                       │      │
│  └───────────┘ └───────────┘ └───────────────────────┘      │
└──────────────────────────────────────────────────────────────┘
                               ↑
┌──────────────────────────────┴──────────────────────────────┐
│                     数据采集层 (Data Collection)              │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐       │
│  │ 定向爬虫  │ │ API 同步  │ │ 文件导入  │ │ 流式数据  │       │
│  │ (Scrapy)  │ │ (REST)    │ │(CSV/PDF)  │ │ (Kafka)   │       │
│  └──────────┘ └──────────┘ └──────────┘ └──────────┘       │
│                      ┌──────────────────┐                    │
│                      │ 数据清洗/标准化   │                    │
│                      └──────────────────┘                    │
└──────────────────────────────────────────────────────────────┘
```

**技术选型：**
- 图数据库：Neo4j 5.x (Bolt 协议)，原生图存储，Cypher 查询语言
- NLP 管线：spaCy 3.8 + LLM-based NER，实体抽取双引擎
- 数据采集：Scrapy + httpx + Playwright，支持分布式采集
- 向量检索：NumPy + scikit-learn (TfIdf/余弦相似度)，轻量级实体对齐
