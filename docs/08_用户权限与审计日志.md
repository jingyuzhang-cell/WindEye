# 08 — 用户权限与审计日志系统

> 版本：v0.4.0 | 最后更新：2026-06-23

---

## 8.1 概述与设计目标

### 8.1.1 背景与动机

WindEye 当前版本（v0.2.0）使用**模拟认证**机制：

- 后端 `POST /api/login/account` 接受任意用户名和密码，返回写死的 `admin` 角色
- 前端 `access.ts` 仅做单一 `canAdmin` 布尔判断
- 无 Token 验证、无会话管理、无权限分级
- 无任何审计日志或 API 调用记录
- 系统唯一数据库为 Neo4j（图数据库），无关系型数据库

`docs/06_非功能性需求.md` 第 6.3.2 节已明确规定：

| 需求项 | 规格 |
|--------|------|
| 身份认证 | JWT Token + Refresh 机制，Token 有效期 ≤ 2h |
| 权限控制 | RBAC，四级角色：admin / analyst / auditor / read-only |
| API 鉴权 | 所有 API 请求携带 Token，中间件统一校验 |
| 审计日志 | 所有数据变更、LLM 调用、用户操作全量记录，不可篡改、不可删除 |
| 日志留存 | ≥ 6 个月 |

上述需求**均未实现**。本文档定义完整的用户管理、权限管理、操作审计和 API 调用日志系统需求。

### 8.1.2 设计目标

| 目标 | 说明 |
|------|------|
| **安全认证** | 基于 JWT 的真实身份认证，替换模拟登录 |
| **细粒度权限** | 页面级 + 按钮级 + API 级 + 数据级四级 RBAC |
| **全量审计** | 操作日志 + API 调用日志双轨记录，不可删除 |
| **可观测性** | 超级管理员首页展示系统运行状态、API 调用趋势、异常告警 |
| **数据隔离** | Neo4j 存储图谱数据，MySQL 存储用户/权限/日志数据，职责清晰 |
| **平滑迁移** | 通过 `AUTH_ENABLED` 开关支持 mock → real auth 渐进切换 |

### 8.1.3 架构定位：MySQL 作为第二数据库

```
┌─────────────────────────────────────────────────┐
│                   WindEye 系统                    │
├─────────────────────┬───────────────────────────┤
│   Neo4j (图数据库)    │   MySQL (关系型数据库)      │
│                     │                           │
│   • 四层知识图谱     │   • 用户账号与密码           │
│   • 实体节点与关系   │   • 角色与权限定义           │
│   • 图查询与分析     │   • 角色-权限关联            │
│   • 风险传导路径     │   • 用户-角色关联            │
│   • 社区发现        │   • 操作审计日志             │
│                     │   • API 调用日志             │
│                     │   • 系统配置                 │
└─────────────────────┴───────────────────────────┘
```

### 8.1.4 数据库职责划分

| 数据类型 | 存储位置 | 原因 |
|----------|----------|------|
| 知识图谱（实体/关系/路径） | Neo4j | 图遍历性能优势 |
| 用户/角色/权限 | MySQL | 结构化数据，关联查询频繁 |
| 操作审计日志 | MySQL | 需要索引、分区、聚合查询 |
| API 调用日志 | MySQL | 高写入量，需要按月分区、定时清理 |
| 系统配置 | MySQL | 键值结构，需要版本化管理 |

### 8.2.4 日志表分区策略

`sys_api_log` 和 `sys_operation_log` 在生产环境应按月进行范围分区：

```sql
ALTER TABLE sys_api_log
  PARTITION BY RANGE (TO_DAYS(created_at)) (
    PARTITION p202601 VALUES LESS THAN (TO_DAYS('2026-02-01')),
    PARTITION p202602 VALUES LESS THAN (TO_DAYS('2026-03-01')),
    -- ... 逐月添加
  );
```

分区带来的优势：
- 按时间范围查询时仅扫描相关分区
- 过期数据可通过 `DROP PARTITION` 快速删除（秒级），无需逐行 DELETE
- 减少锁竞争，提升并发写入性能

## 8.3 用户管理

### 8.3.1 用户 CRUD API

| 方法 | 端点 | 权限 | 说明 |
|------|------|------|------|
| GET | `/api/v1/admin/users` | `admin:user:view` | 分页用户列表，支持搜索和状态筛选 |
| POST | `/api/v1/admin/users` | `admin:user:create` | 创建用户 |
| GET | `/api/v1/admin/users/{user_id}` | `admin:user:view` | 查看用户详情（含角色） |
| PUT | `/api/v1/admin/users/{user_id}` | `admin:user:update` | 修改用户信息 |
| PATCH | `/api/v1/admin/users/{user_id}/status` | `admin:user:disable` | 启用/禁用/解锁用户 |
| POST | `/api/v1/admin/users/{user_id}/reset-password` | `admin:user:create` | 管理员重置用户密码 |
| DELETE | `/api/v1/admin/users/{user_id}` | `admin:user:delete` | 软删除用户（设置 deleted=1） |

### 8.3.2 密码策略

| 策略项 | 值 | 说明 |
|--------|-----|------|
| 哈希算法 | bcrypt (cost=12) | 使用官方 `bcrypt` 包 |
| 最小长度 | 8 字符 | 可通过 sys_config 调整 |
| 最大长度 | 64 字符 | — |
| 复杂度要求 | 至少包含 2 类（大写/小写/数字/特殊字符） | — |
| 禁止包含 | 用户名 | 大小写不敏感 |
| 历史检查 | 最近 5 次密码不可重复 | 存储在 `sys_password_history` 或专用表 |
| 过期时间 | 90 天 | 0 = 永不过期 |
| 过期提醒 | 提前 7 天 | 登录时提示修改 |
| 锁定策略 | 连续 5 次失败 → 锁定 30 分钟 | 自动解锁 `locked_until` |
| 管理员重置 | 管理员重置后强制下次登录修改密码 | `password_updated_at` 设为 NULL |

### 8.3.3 账号状态

| 状态码 | 含义 | 触发条件 |
|--------|------|----------|
| `1` — active | 正常 | 创建默认、管理员解锁 |
| `0` — disabled | 禁用 | 管理员手动禁用 |
| `2` — locked | 锁定 | 连续登录失败达到上限，自动解除 |

## 8.4 权限管理 (RBAC)

### 8.4.1 四角色定义

| 角色编码 | 角色名称 | 权限范围 |
|----------|----------|----------|
| `admin` | 管理员 | 全部系统权限：用户管理、角色管理、权限分配、系统配置、所有图谱操作、审计日志查看 |
| `analyst` | 分析师 | 图谱查询、风险分析、报告生成、数据上传、图谱导出。不可管理用户和配置 |
| `auditor` | 审核员 | 只读访问全部数据 + 操作日志 + API 日志。不可修改、导出、管理用户 |
| `readonly` | 只读用户 | 只读访问图谱查询和报告。不可导出、不可看日志、不可管理任何内容 |

### 8.4.2 四级权限粒度

| 粒度 | 说明 | 示例 |
|------|------|------|
| **页面级 (page)** | 能否访问某个前端页面 | `page:admin:view` → 可见 `/system/admin` 菜单 |
| **按钮级 (button)** | 能否点击某个操作按钮 | `graph:search:export` → 显示 PNG/CSV 导出按钮 |
| **API 级 (api)** | 能否调用某个后端接口 | `api:user:create` → 可调 `POST /api/v1/admin/users` |
| **数据级 (data)** | 数据的可见范围 | `data:report:all` → 查看所有报告；否则仅看自己创建的 |

### 8.4.3 权限编码规范

格式：`{module}:{resource}:{action}`

| 模块 (module) | 资源 (resource) | 动作 (action) |
|---------------|-----------------|---------------|
| `system` — 系统管理 | `monitor` | `view` |
| `admin` — 用户管理 | `user` | `create` / `update` / `disable` / `delete` |
| `admin` — 角色管理 | `role` | `view` / `assign` |
| `graph` — 图谱 | `search` / `expand` | `view` / `export` |

### 8.4.5 权限检查流程

```
用户请求 (含 JWT Token)
  │
  ▼
AuthMiddleware
  ├─ 解码 JWT，提取 user_id
  ├─ 检查 Token 是否在黑名单 (Redis)
  └─ 设置 request.state.user_id
  │
  ▼
PermissionMiddleware
  ├─ 查找 Redis 缓存: user:{user_id}:permissions
  ├─ 缓存命中 → 直接校验
  └─ 缓存未命中 → 查 MySQL:
       SELECT p.perm_code
       FROM sys_permission p
       JOIN sys_role_permission rp ON p.id = rp.permission_id
       JOIN sys_user_role ur ON rp.role_id = ur.role_id
       WHERE ur.user_id = ? AND p.perm_type IN ('api', 'data') AND p.status = 1
     → 写入 Redis (TTL 300s)
  │
  ├─ 请求所需权限 ∈ 用户权限集合 → 放行
  └─ 不在 → 返回 403
```

|------|------|

### 8.6.3 日志脱敏策略

**敏感字段清单（禁止明文存入日志）：**

```
password
token
accessToken
refreshToken
authorization
phone           (保留前3后4, 中间打码：138****8000)
email           (保留首字符，其余打码：z****@example.com)
idCard          (保留前3后4)
newPassword
oldPassword
confirmPassword
secret
apiKey
```

**脱敏函数：**

```python
import re

_SENSITIVE_KEYS = {
    "password", "token", "accesstoken", "refreshtoken",
    "authorization", "phone", "email", "idcard", "id_card",
    "newpassword", "oldpassword", "confirmpassword", "secret", "apikey",
}

def mask_sensitive_data(data: dict | list | None) -> dict | list | None:
    """递归脱敏：将 dict 中所有敏感字段替换为 '***MASKED***'。

    对于 phone/email/idCard 等可识别字段，保留格式提示并部分打码。
    """
    if data is None:
        return None

    if isinstance(data, list):
        return [mask_sensitive_data(item) for item in data]

    if isinstance(data, dict):
        result = {}
        for key, value in data.items():
            key_lower = key.lower().replace("_", "").replace("-", "")
            if key_lower in _SENSITIVE_KEYS:
                result[key] = "***MASKED***"
            elif isinstance(value, (dict, list)):
                result[key] = mask_sensitive_data(value)
            else:
                result[key] = value
        return result

    return data
```

@app.middleware("http")
async def api_log_middleware(request: Request, call_next):
    # 跳过自我监控路径
    if request.url.path in EXCLUDE_PATHS:
        return await call_next(request)

    start = time.perf_counter()
    response = await call_next(request)
    duration_ms = int((time.perf_counter() - start) * 1000)

    # 异步写入，不阻塞响应
    background_tasks.add_task(
        api_log_service.write,
        trace_id=get_trace_id(),
        user_id=getattr(request.state, 'user_id', None),
        method=request.method,
        path=request.url.path,
        status_code=response.status_code,
        latency_ms=duration_ms,
        ...
    )

### 8.8.5 JWT 拦截器

```typescript

// 响应拦截器：401 自动刷新 Token 或跳转登录
responseInterceptors: [
  async (response: Response) => {
    if (response.status === 401) {
      // 尝试使用 refreshToken 刷新
      const refreshed = await tryRefreshToken();
      if (!refreshed) {
        localStorage.removeItem('accessToken');
        localStorage.removeItem('refreshToken');
        window.location.href = '/user/login';
      }
    }
    if (response.status === 403) {
      message.error('无权限执行此操作');
    }
    return response;
  },
],
```

| # | 标准 | 验证方式 |
|---|------|----------|
| 1 | 系统不再使用任意用户名密码登录 | 尝试任意用户名密码登录 → 返回 401 |

### 8.12.2 总体实现架构

```text
浏览器
  ├─ AuthStore：内存 Access Token、当前用户、权限集合
  ├─ RequestClient：附加 Token、单飞刷新、401/403 处理
  ├─ Route Access：页面级权限
  └─ Permission/Access：按钮级权限
        │
        ▼
FastAPI
  ├─ TraceIdMiddleware
  ├─ ApiLogMiddleware（覆盖 401/403/5xx）
  ├─ AuthMiddleware（解析身份）
  ├─ require_permissions()（路由依赖，执行授权）
  ├─ Auth/Admin/Audit/Dashboard Routers
  └─ Service + Unit of Work
        │
        ├─ MySQL：用户、角色、权限、会话、操作日志、API 日志
        ├─ Redis：会话撤销、权限缓存、刷新锁
        └─ Neo4j：图谱与报告；写入 created_by 以支持数据权限
```

关键原则：

1. **认证与授权分离**：中间件只负责解析身份；每个受保护端点通过 `require_permissions()` 显式声明权限码。
2. **后端权限为最终边界**：前端隐藏菜单和按钮仅改善体验，不能代替后端 403 校验。
3. **操作日志与业务变更同事务**：用户、角色、权限修改成功时，业务数据和操作日志一次提交。
4. **API 日志异步批量写入**：允许短暂延迟，但不得无限创建后台任务或影响主请求。
5. **默认拒绝**：`AUTH_ENABLED=true` 时，未显式标记为公开的端点全部要求认证。

灰度期间建议新增 `AUTH_MODE=off|observe|enforce`，逐步替代布尔型 `AUTH_ENABLED`：

- `off`：开发兼容模式，不解析身份、不执行权限判断。
- `observe`：解析身份并记录授权结果，但暂不返回 403。
- `enforce`：正式模式，认证或授权失败立即返回 401/403。

### 8.13.2 认证流程

#### 登录

1. 按用户名读取未删除用户。
2. 检查禁用、锁定和密码过期状态。
3. bcrypt 校验密码，失败时累计次数并记录 `LOGIN_FAILED`。
4. 成功后创建 `sys_auth_session`。
5. Access Token 包含 `sub`、`sid`、`jti`、`token_version`、`iat`、`exp`、`iss`、`aud`、`type=access`。
6. Refresh Token 仅保存哈希，明文通过 `HttpOnly + Secure + SameSite=Lax` Cookie 返回。
7. 返回 Access Token、过期时间和当前用户摘要。

生产约束：

- `JWT_SECRET` 未配置或长度不足 32 字节时，应用启动失败，不允许使用开发兜底密钥。
- 种子脚本不再创建 `admin/admin`；首次管理员密码来自一次性环境变量，首次登录必须修改。
- 登录错误统一返回“用户名或密码错误”，具体禁用/锁定原因只写审计日志。
- 登录接口增加用户名/IP 双维度限流。
- Refresh Cookie 接口校验 `Origin`/`Referer` 和自定义请求头，降低跨站请求风险。

#### 刷新

Refresh Token 采用轮换机制：每次刷新撤销旧会话令牌并签发新令牌；检测到已使用令牌重放时，撤销同一会话链。前端并发出现多个 401 时只允许发起一个刷新请求，其余请求等待同一 Promise。

#### 登出与强制失效

- 普通登出：撤销当前 `sid`。
- 修改密码或管理员重置密码：`token_version + 1`，撤销该用户全部会话。
- 禁用或删除用户：同样提升 `token_version` 并撤销全部会话。
- Redis 不可用时不得静默放行已撤销会话；回退到 MySQL 会话表校验并产生告警。

权限集合查询规则：

1. 仅加载启用用户、启用角色、启用权限。
2. Redis Key 使用 `rbac:user:{user_id}:v{permission_version}`，TTL 建议 5 分钟。
3. 修改用户角色或角色权限后，提交事务再删除相关缓存。
4. Redis 故障时回源 MySQL，不得把故障解释为“拥有全部权限”。
5. `sys_permission.resource_path` 用于管理界面展示和审计，不作为唯一运行时授权来源；接口代码中的权限声明才是可审查的安全边界。

- `data:report:all`：可查看全部报告。
- `data:report:own`：Neo4j `RiskReport.created_by` 必须等于当前用户 ID。
- 创建报告、导出报告和删除报告时均传递 `Principal`，禁止信任前端提交的 `created_by`。

### 8.13.4 事务与审计一致性

现有 service 内部多次 `commit()` 会造成“业务成功但日志失败”或“日志成功但业务回滚”。应调整为：

```text
Route/UnitOfWork
  begin
    service 修改业务数据，只 flush，不 commit
    audit_service.add_operation_log，同一 session 写入
  commit
```

操作失败时：

- 可预期业务失败记录 `result=FAILURE`，不保存密码、Token、文件正文等敏感内容。
- 数据库整体不可用时，业务变更失败；登录失败日志可写入安全的结构化应用日志作为降级证据。

审计“不可篡改”不能仅靠“不提供删除 API”实现，生产环境至少采用：

1. 应用数据库账号对审计表只有 `INSERT/SELECT`，无 `UPDATE/DELETE`。
2. 清理任务使用独立维护账号，权限和凭据分离。
3. 每条记录保存 `payload_hash`，每日生成 `sys_audit_seal.root_hash`。
4. 每日封存结果导出到启用对象锁的归档存储。
5. `sys_operation_log` 默认长期保留；`sys_api_log` 满 180 天后先归档再删除分区。

### 8.13.5 API 日志可靠写入

用应用级有界队列替换逐请求 `asyncio.create_task()`：

```text
请求结束 → 写入 asyncio.Queue(maxsize=N)
                    ↓
           单后台 Worker 批量写 MySQL
                    ↓
       应用关闭前 drain，超时后记录丢弃计数
```

首期建议每 100 条或每 500ms 批量提交。队列满时不阻塞业务，但必须增加 `api_log_dropped_total` 指标和 ERROR 告警。后续高并发部署可替换为 Redis Stream/Kafka，接口层保持不变。

中间件运行顺序必须通过集成测试验证，确保 API 日志能看到 401、403 和 500，Trace ID 同时返回响应头并写入两类日志。

### 8.14.2 登录与请求链路

改造顺序：

1. 将 `frontend/src/services/ant-design-pro/api.ts` 的登录、当前用户和登出端点切换到 `/api/v1/auth/*`。
2. 登录成功后保存 Access Token；Refresh Token 由浏览器 Cookie 管理。
3. `getInitialState()` 调用 `/api/v1/auth/me`，得到 `roles` 和 `permissions`。
4. 请求拦截器附加 `Authorization: Bearer <accessToken>`，删除当前 `?token=123` 示例逻辑。
5. 响应遇到 401 时单飞刷新并只重试原请求一次；刷新失败后清理身份并跳转登录页。
6. 403 显示明确提示，但不清除登录状态。
7. 登出先调用后端撤销会话，再清理前端状态。

Access Token 推荐仅保存在内存；页面刷新后通过 Refresh Cookie 恢复会话。若首期必须降低实现复杂度，可临时使用 `sessionStorage`，不得把 Refresh Token 放入 `localStorage`。

### 8.14.3 页面级与按钮级权限

`frontend/src/access.ts` 改为：

```typescript
export default function access(initialState?: InitialState) {
  const permissions = new Set(initialState?.currentUser?.permissions ?? []);
  const has = (code: string) => permissions.has(code);

  return {
    isAuthenticated: Boolean(initialState?.currentUser),
    canViewSystem: has('system:monitor:view'),
    canManageUsers: has('admin:user:view'),
    canManageRoles: has('admin:role:view'),
    canViewAuditLogs: has('audit:operation-log:view'),
    canViewApiLogs: has('audit:api-log:view'),
  };
}
```

按钮权限统一使用组件，不在页面中散落字符串判断：

```tsx
<Permission code="admin:user:create">
  <Button type="primary" icon={<PlusOutlined />}>新建用户</Button>
</Permission>
```

无页面权限时路由显示 403；无按钮权限时默认隐藏。对“查看详情”等需要解释的场景可禁用并提供 Tooltip。

## 8.17 上线切换方案

1. 在测试环境部署 MySQL、Redis，执行迁移和安全种子脚本。
2. 保持 `AUTH_ENABLED=false`，先联调真实认证端点和管理页面。
3. 开启“仅记录不拦截”权限观测模式，统计现有接口缺失的权限声明。
4. 补齐权限映射后，在测试环境设置 `AUTH_ENABLED=true`。
5. 使用四类角色执行回归测试和前端浏览器验收。
6. 生产发布前强制检查 `JWT_SECRET`、初始管理员、数据库最小权限和归档任务。
7. 生产灰度开启认证；保留快速回退配置，但回退必须产生安全告警和审计记录。
8. 稳定一个发布周期后删除 `/api/login/account`、`/api/currentUser` 等 mock 兼容端点。

最终完成标准：前后端不再依赖 mock 身份；每个受保护 API 都有明确权限码；所有关键操作可按用户和 Trace ID 追溯；禁用账号、权限变更和会话撤销能够即时生效。
