# 04 — 多元协同治理

## 4.1 模块概述

多元协同治理是 BiDA-KG 的**核心大脑**。利用大模型（LLM）编排多个专属智能体（Planner → Retriever → Analyst → Compliance → Reporter），基于四层知识图谱进行风险传递推理、异常发现和合规研判，最终生成协同治理社区报告。采用 **DRA-MA (Dynamic Reasoning & Multi-Agent)** 架构，支持灵活的角色定义和协同编排。

整体链路：**风险主体识别 → 群体发现 → 风险传导路径分析 → 合规匹配 → 协同治理社区报告**

---

## 4.2 智能体角色定义

### 4.2.1 Planner Agent (规划者)

| 属性 | 说明 |
|------|------|
| **职责** | 接收用户问题，拆解为子任务，编排执行计划，分配给各 Agent |
| **输入** | 自然语言问题 / 系统触发信号 |
| **输出** | 结构化任务计划 (Task Plan) |
| **核心 Prompt** | "你是一个风险分析任务规划专家。请将用户的查询请求拆解为可执行的子任务序列，并指定每个子任务最适合执行的 Agent。" |

### 4.2.2 Retriever Agent (检索者)

| 属性 | 说明 |
|------|------|
| **职责** | 将自然语言信息需求转化为 Cypher 查询，从 Neo4j 知识图谱中提取相关子图 |
| **输入** | Planner 分配的信息需求描述 |
| **输出** | 结构化的子图数据（节点列表 + 关系列表） |
| **核心能力** | Text-to-Cypher：自然语言 → Cypher 查询语句。支持：多跳路径查询、子图扩展、属性过滤、聚合统计 |

**安全约束：** Retriever Agent 生成的 Cypher 必须是**只读**的 (MATCH / RETURN)，禁止 CREATE / DELETE / SET / MERGE 操作。

### 4.2.3 Analyst Agent (分析者)

| 属性 | 说明 |
|------|------|
| **职责** | 接收子图数据，结合内置风控逻辑和推理能力，分析风险传递路径和异常模式 |
| **输入** | Retriever 返回的子图数据 + 上下文 |
| **输出** | 风险分析结论：风险路径、异常节点、风险等级评分、关键证据 |

**分析维度：**

| 维度 | 分析内容 | 输出 |
|------|---------|------|
| **股权传导** | A → 控股 → B → 控股 → C，A 暴雷时 B、C 受影响程度 | 传导路径 + 影响权重 |
| **人事关联** | A 公司高管同时任职于 B 公司，风险通过人员传导 | 关键人物 + 关联企业 |
| **担保链** | 担保关系形成的风险传递链，识别担保圈 | 担保网络 + 脆弱节点 |
| **资金轨迹** | 异常资金流动路径分析 | 资金路径 + 可疑标记 |
| **事件因果** | A 事件 → B 事件 → C 事件的因果链推理 | 事件因果图 |

### 4.2.4 Compliance Agent (合规者)

| 属性 | 说明 |
|------|------|
| **职责** | 审核分析 Agent 的结论，匹配法规/行动层知识，输出违规认定和法律依据 |
| **输入** | Analyst 的分析结论 + 法规层数据 |
| **输出** | 违规认定、匹配法条、建议处置动作、合规风险等级 |

**匹配流程：**
```
分析结论中的风险行为 → 检索法规层 → 匹配适用法条 →
提取违规要件 → 逐一比对 → 输出合规判定结果
```

### 4.2.5 Reporter Agent (汇报者)

| 属性 | 说明 |
|------|------|
| **职责** | 汇总所有 Agent 的输出，生成结构化的风险分析报告 |
| **输入** | 所有 Agent 的中间结论 + 子图数据 |
| **输出** | Markdown/JSON 格式的结构化风险报告 |

## 4.3 协同编排引擎 (Orchestrator)

### 4.3.1 编排模式

| 模式 | 描述 | 适用场景 |
|------|------|---------|
| **Sequential** (顺序) | Planner → Retriever → Analyst → Compliance → Reporter，线性流水线 | 标准风险分析任务 |
| **Debate** (辩论) | 两个 Analyst Agent 独立分析后交叉质证，取共识结论 | 高争议性风险研判 |
| **Reflection** (反思) | Agent 输出经自检后迭代优化，直至质量达标 | 复杂报告生成 |
| **HITL** (人在回路) | 在关键节点暂停，待业务专家审核后再继续 | 高风险等级任务 |

### 4.3.2 编排流程 (Sequential 模式)

```
            ┌─────────┐
            │  用户    │ 自然语言查询 / 事件触发
            └────┬────┘
                 │
        ┌────────▼────────┐
        │  Planner Agent   │ 拆解任务 → 生成执行计划
        └────────┬────────┘
                 │
        ┌────────▼────────┐
        │ Retriever Agent  │ 计划[N] → Cypher → Neo4j → 子图数据
        └────────┬────────┘
                 │
        ┌────────▼────────┐
        │ Analyst Agent    │ 子图数据 → 风险路径分析 → 异常标记
        └────────┬────────┘
                 │
        ┌────────▼────────┐      ┌─────────────────┐
        │ Compliance Agent │ ←── │ 法规层 KG 数据   │
        └────────┬────────┘      └─────────────────┘
                 │
        ┌────────▼────────┐
        │ Reporter Agent   │ 汇总 → 结构化报告
        └────────┬────────┘
                 │
        ┌────────▼────────┐
        │ Report Review    │ 高风险 → 报告复核 → 修正
        └────────┬────────┘
                 │
        ┌────────▼────────┐
        │  业务应用层      │ 报告展示 / 合规分析
        └─────────────────┘
```

### 4.3.3 报告复核 (Report Review)

**介入点：**
1. Planner 生成计划后 — "是否同意此分析计划？"
2. Analyst 得出结论后 — "请确认风险分析结论是否合理"
3. Reporter 生成报告后 — "请审核报告内容，可修改后确认发布"

**干预类型：**
- **修正：** 修改 Agent 的错误结论或参数
- **补充：** 添加 Agent 遗漏的关键信息
- **驳回：** 推翻结论，要求重新分析（带反馈）

## 4.4 异常发现

### 4.4.1 主动异常检测

系统定期（每日/每小时）对全图或特定子图运行异常检测算法：

| 检测类型 | 算法 | 说明 |
|---------|------|------|
| **资金环流检测** | 有向环检测 (DFS/Tarjan) | 识别 A→B→C→A 的循环资金流 |
| **社区欺诈检测** | Louvain 社区发现 + 密度分析 | 发现紧密耦合的异常关联群 |
| **突发实体检测** | 时序分析：节点度突变 | 某实体短期内关联关系剧增 |
| **结构洞检测** | Betweenness Centrality | 识别网络中关键"桥梁"节点 |
| **异常路径检测** | LLM-based 模式匹配 | 用自然语言描述异常模式，在图中搜索匹配 |

### 4.4.2 触发式分析

当系统监测到以下事件时，自动触发多智能体协同分析：

- 图谱中新增"被执行"、"失信"、"破产"类事件节点
- 某重点监控实体发生属性重大变更（法人/股权/经营范围）
- 负面舆情事件达预设热度阈值
- 监管机构发布新的风险提示或法规

## 4.5 LLM 集成规范

### 4.5.1 模型配置

```env
LLM_API_KEY=sk-xxx          # API 密钥
LLM_BASE_URL=https://api.deepseek.com/v1  # Base URL
LLM_MODEL=deepseek-v4-flash  # 模型名称
```

### 4.5.2 调用策略

| 场景 | Temperature | 原因 |
|------|-------------|------|
| Text-to-Cypher | 0.0 | 需要确定性输出 |
| 风险分析 | 0.3 | 需要一定创造性但保持可控 |
| 报告撰写 | 0.5 | 需要流畅自然的语言表达 |
| 实体抽取 (NER) | 0.0 | 需要精确的结构化输出 |

### 4.5.3 私有化部署要求

- 生产环境中 LLM 应私有化部署，确保敏感数据不出域
- 调用外部 API 时，输入数据需经过去敏处理（企业名称可用代号替代，个人姓名/证件号脱敏）
- 所有 LLM 调用需记录日志，包括请求时间、Prompt 摘要、Token 消耗、响应耗时

## 4.6 协同治理统一管线 (UnifiedEngine)

协同治理模块的实际运行入口是 `POST /api/v1/chat/unified-stream`，通过 SSE 事件流依次推送完整的分析和治理结果。

### 4.6.1 统一管线阶段

```
用户输入 query
    │
    ▼
[IntentAgent] 意图识别 → 实体抽取 → 意图分类 (graph_qa / risk_analysis)
    │
    ▼
[EntityResolver] 实体对齐 → 5级级联匹配 (精确→别名→包含→模糊→LLM回退)
    │
    ▼
[DRAEngine] 子图查询 → 实体统计 → 群体发现 (WCC/Louvain) → PageRank 中心性
    │
    ▼
[Risk Plugins] 5个风险插件依次执行:
    ├── RiskAnalyst: 风险路径识别 + 异常发现
    ├── Compliance: 法规匹配 + 合规评分
    ├── Scoring: 6维风险评分 (关系复杂度/风险关系/社区密度/传导/合规/证据)
    ├── Governance: 治理动作 + 升级规则 + 监控清单
    └── Reporter: 执行摘要 + Markdown报告 + 建议措施
```

### 4.6.2 SSE 事件类型 (16种)

| 事件 | 阶段 | 说明 |
|------|------|------|
| `stage` | 全局 | 当前阶段变更 (intent/resolve/retrieve/analyze/compliance/scoring/governance/report) |
| `entities` | 实体解析 | 已对齐实体 + 未解析实体 |
| `subgraph` | 图谱检索 | 证据子图 nodes + edges |
| `entity_stats` | 图谱分析 | 实体统计信息 |
| `community` | 群体发现 | 社区列表、成员数、算法 |
| `entity_community_map` | 群体发现 | 实体→社区归属 + 角色 (bridge/hub/member) |
| `candidate_risk_paths` | 风险分析 | 候选风险路径 |
| `risk_paths` | 风险分析 | 确认的风险路径 + 描述 + 置信度 |
| `anomaly_findings` | 风险分析 | 异常发现列表 |
| `compliance` | 合规分析 | 法规匹配结果 |
| `compliance_scores` | 合规分析 | 合规指标得分 |
| `compliance_indicators` | 合规分析 | 一/二/三级合规指标详情 |
| `scoring` | 风险评分 | 6维评分 + LLM调整 |
| `governance` | 治理方案 | 治理动作 + 升级规则 + 监控清单 |
| `report` | 报告生成 | executive_summary + markdown_report + recommendations |
| `done` / `error` | 终态 | 完成或错误 |

## 4.7 协同治理社区报告

### 4.7.1 报告结构

```
风险主体识别 → 群体发现 → 风险传导路径 → 合规分析 → 协同治理社区报告
```

### 4.7.2 报告字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `report_id` | String | 唯一报告编号 |
| `executive_summary` | String | 社区治理结论摘要 |
| `related_entities` | Array[Entity] | 关联风险主体 |
| `communities` | Array[Community] | 群体发现结果 |
| `risk_paths` | Array[Path] | 风险传导路径 |
| `compliance_matches` | Array[Compliance] | 合规匹配结果 |
| `governance_plan` | Object | 治理建议、优先级和升级规则 |
| `recommendations` | Array[String] | 建议措施 |
| `created_at` | DateTime | 生成时间 |

### 4.7.3 治理建议类型

| 建议类型 | 说明 |
|---------|------|
| **风险核查** | 对关键主体、资金链路和股权关系进行重点核查 |
| **合规补充** | 补充披露、关联关系说明或内部决策材料 |
| **持续观察** | 跟踪风险传导路径上的主体状态变化 |
| **升级研判** | 对高风险路径触发进一步研判和人工复核 |

### 4.7.4 报告导出

- **在线查看：** 浏览器中渲染的 HTML 报告 (ProLayout + 组件化)
- **导出格式：** PDF (Puppeteer 服务端渲染) / Word (docx 模板) / Markdown

## 4.8 风险可视化看板

基于 AntV G6 的知识图谱可视化看板，支持分层展示、交互式探查和风险路径追踪。

### 4.8.1 分层展示

```
┌──────────────────────────────────────────┐
│  [图层切换]  ☑ 主体层  ☑ 事件层          │
│              ☑ 特征层  ☑ 法规层          │
├──────────────────────────────────────────┤
│        ┌──────────┐                      │
│        │ 法规节点  │  ← 第4层 (顶部)      │
│        └─────┬────┘                      │
│              │ 关系边                     │
│        ┌─────┴────┐                      │
│        │ 特征节点  │  ← 第3层             │
│        └─────┬────┘                      │
│              │                           │
│        ┌─────┴────┐                      │
│        │ 事件节点  │  ← 第2层             │
│        └─────┬────┘                      │
│              │                           │
│        ┌─────┴────┐                      │
│        │ 主体节点  │  ← 第1层 (底部)      │
│        └──────────┘                      │
└──────────────────────────────────────────┘
```

### 4.8.2 交互操作

| 操作 | 行为 | 技术要求 |
|------|------|---------|
| **节点点击** | 展开节点详情面板 (属性列表 + 1跳邻居) | < 200ms 响应 |
| **节点下钻** | 以当前节点为中心，展开 N 跳邻居 | 默认 1 跳，可配置 2-3 跳 |
| **关系高亮** | 鼠标悬停关系边，高亮显示关联节点和路径 | 高亮 + 其他节点半透明 |
| **路径追溯** | 选中两个节点，自动查询并高亮所有路径 | 调用 `/kg/path` API |
| **框选/圈选** | 批量选中节点，支持批量标记/导出 | G6 框选交互 |
| **子图导出** | 将当前可视区域导出为 PNG/SVG 图片 | 用于嵌入报告 |
| **风险滤镜** | 按风险等级筛选节点 (高/中/低/无) | 颜色 + 筛选联动 |
| **全文搜索** | 搜索框输入实体名称，自动定位 | 模糊匹配 + 自动居中 |

### 4.8.3 画布控制

- 缩放 (0.1x ~ 5x)
- 全图适配 (FitView)
- 撤销/重做 (操作历史栈，深度 20)
- 布局切换 (力导向 / 分层 / 环形 / 网格 / 同心圆)

## 4.9 知识图谱风险问答 (KnowledgeQA)

面向终端用户的智能交互入口。用户通过自然语言提出风险相关问题，后端调用 DRA-MA 多智能体架构进行协同推理，前端**实时展示推理过程**、生成关联知识图谱并提供数据分析能力。

### 4.9.1 核心价值

| 维度 | 描述 |
|------|------|
| **交互自然化** | 支持中文自然语言输入，用户无需掌握 Cypher 或图查询语言 |
| **推理透明化** | 实时流式展示多智能体思考过程 |
| **结果可视化** | 推理结果自动关联知识图谱子图，以 G6 力图展示实体与关系网络 |
| **分析深度化** | 支持路径追踪、异常发现、法规匹配，覆盖股权传导、人事关联、担保链、资金轨迹、事件因果 |

### 4.9.2 前端架构

采用 **双 Zustand Store** 模式：

```
┌─────────────────────────────────────┐
│         agentStore (内存)            │
│  当前会话工作状态:                    │
│  - messages[] / currentSubgraph     │
│  - analysisResult / currentRoute    │
│  - riskScores / governancePlan      │
│  - agentTraces (调试)               │
│  方法: sendMessage(), clearHistory() │
└──────────────┬──────────────────────┘
               │ auto-save (useEffect debounce 1s)
               ▼
┌─────────────────────────────────────┐
│     chatStore (localStorage 持久化)  │
│  多会话持久化:                       │
│  - sessions[] / activeSessionId     │
│  方法: createNewSession(),           │
│        switchSession(),              │
│        deleteSession()              │
└─────────────────────────────────────┘
```

### 4.9.3 界面布局

三栏式布局：左侧栏（会话管理 260px）+ 中间栏（聊天窗口 480px）+ 右侧栏（图谱视图 / 治理报告 / 分析面板）

右侧面板通过 Segmented 切换三种视图：
- **图谱视图：** G6 力导向图 + 社区着色 + 图例面板
- **治理报告：** 风险评分详情 + 风险路径 + 综合报告 + 协同治理方案
- **分析面板：** ECharts 图表 + 数据预览表格

### 4.9.4 上下文管理

- 双击图谱节点将实体注入聊天上下文（`[EntityName]` 前缀）
- ContextTagBar 水平标签栏，支持删除单个/清空全部
- 上限 5 个实体标签，超出显示 "+N more"
