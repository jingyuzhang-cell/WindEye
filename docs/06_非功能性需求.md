# 06 — 非功能性需求

## 6.1 性能要求

### 6.1.1 响应时间

| 操作 | 目标 (P95) | 可接受上限 (P99) |
|------|-----------|-----------------|
| 前端页面首次加载 (FCP) | < 2s | < 4s |
| KG 单实体查询 (1跳) | < 200ms | < 500ms |
| KG 多跳扩展 (3跳) | < 500ms | < 1s |
| KG 路径查询 (5跳) | < 1s | < 2s |
| KG 全文搜索 | < 300ms | < 800ms |
| 图可视化渲染 (1000节点) | < 2s | < 5s |
| 图可视化渲染 (5000节点) | < 5s | < 10s |
| Multi-Agent 全流程分析 | < 60s | < 120s |
| 报告生成 (含图表) | < 30s | < 60s |
| 数据采集-入库延迟 | < 1h (常规) | < 24h (全量) |

## 6.3 安全要求

### 6.3.1 数据安全

| 要求 | 具体措施 |
|------|---------|
| **传输加密** | 全链路 HTTPS/TLS 1.3，Neo4j Bolt 加密 |
| **存储加密** | 数据库落盘加密 (AES-256)，个人敏感字段加密存储 |
| **数据脱敏** | 调外部 LLM API 时自动脱敏：企业名→代号，人名/证件号→哈希 |
| **数据隔离** | 多租户数据隔离 (如需)，按用户角色控制数据可见范围 |
| **数据不出域** | 生产环境 LLM 必须私有化部署，图谱数据不离开内网 |

### 6.3.2 访问控制

| 要求 | 具体措施 |
|------|---------|
| **认证** | JWT Token + 刷新机制，Token 有效期 ≤ 2h |
| **授权** | RBAC (基于角色的访问控制)，四级权限：管理员/分析师/审核员/只读 |
| **API 鉴权** | 所有 API 请求携带 Token，中间件统一校验 |
| **Cypher 注入防护** | Retriever Agent 生成 Cypher 仅允许只读操作 (MATCH/RETURN)，禁止写入 |
| **审计日志** | 所有数据变更、LLM 调用、用户操作全量记录，不可篡改、不可删除 |

### 6.3.3 合规要求

- 符合《数据安全法》《个人信息保护法》的数据处理规范
- 个人金融信息保护 (如适用：《个人金融信息保护技术规范》)
- 日志留存 ≥ 6 个月，支持按需导出

## 6.4 扩展性要求

### 6.4.1 功能扩展

| 扩展点 | 说明 |
|--------|------|
| **新 Agent 角色** | 支持通过配置文件注册新 Agent，定义其 Prompt、工具集、输入输出 Schema |
| **新数据源** | 数据采集模块支持插件化添加新数据源适配器 |
| **新本体类型** | 知识图谱本体模型支持通过配置扩展新的实体类型和关系类型 |
| **新分析模式** | Agent 编排引擎支持注册自定义工作流 (Workflow as Code) |

### 6.4.2 技术扩展

| 扩展点 | 说明 |
|--------|------|
| **水平扩展** | API 服务无状态，支持多实例负载均衡部署 |
| **图数据库扩展** | Neo4j 集群 (Causal Cluster) 或 NebulaGraph 分布式扩展 |
| **LLM 模型切换** | 模型配置化，支持 OpenAI / DeepSeek / 私有模型无缝切换 |
| **消息队列** | 采集任务和 Agent 异步任务通过消息队列解耦 (Redis / RabbitMQ) |

## 6.6 可观测性要求

| 维度 | 要求 |
|------|------|
| **日志** | 结构化日志 (JSON)，包含 trace_id 用于全链路追踪 |
| **监控** | Prometheus + Grafana：API QPS/延迟/错误率、KG 规模、Agent 调用量、采集任务状态 |
| **告警** | 关键指标异常 (P99延迟超阈值、错误率飙升、采集任务失败) 实时推送 (钉钉/企微/邮件) |
| **链路追踪** | OpenTelemetry，覆盖 API → Agent 编排 → LLM 调用 → Neo4j 查询全链路 |

## 6.7 CI/CD 要求

| 要求 | 说明 |
|------|------|
| **代码管理** | Git，单一主干开发 (Trunk-Based Development)，统一在 `main` 或约定主干分支上提交 |
| **代码规范** | Biome (前端) + Ruff (后端)，提交前自动检查 |
| **自动化测试** | 单元测试覆盖率 ≥ 70% (核心模块 ≥ 85%) |
| **构建** | 前端：Mako (Rust)；后端：Docker 镜像 |
| **部署** | Docker Compose (开发/测试环境)，Kubernetes (生产环境) |
